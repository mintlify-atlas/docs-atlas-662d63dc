Multiagente is a multi-agent orchestration system built with the Botpress ADK (TypeScript). It receives messages via chat and webchat, delegates tasks to domain specialist agents — HR, IT, Sales, Finance, and Docs — and seamlessly escalates to human support when needed through a HITL (human-in-the-loop) flow.

## How it works

Multiagente acts as a central orchestrator. Every incoming message passes through a single conversation handler that decides whether to respond directly, delegate to a specialist, or escalate to a human.

<Steps>
  <Step title="Message arrives">
    A user sends a message via `chat` or `webchat` channel. The orchestrator receives it and checks if HITL mode is active.
  </Step>
  <Step title="Route to specialist">
    The orchestrator AI reads each specialist's description and delegates the task — HR, IT, Sales, Finance, or Docs — by invoking that agent as a tool.
  </Step>
  <Step title="Specialist executes in isolation">
    Each specialist runs in worker mode with its own context window, calls its domain tools, and returns a structured result to the orchestrator.
  </Step>
  <Step title="Orchestrator responds">
    The orchestrator synthesizes the specialist's result into a single, coherent reply sent back to the user.
  </Step>
</Steps>

## Key features

<CardGroup cols={2}>
  <Card title="Isolated worker agents" icon="box-open">
    Specialists run in their own `execute()` loop with no access to the conversation transcript, keeping the orchestrator in full control.
  </Card>
  <Card title="Human-in-the-loop" icon="person-circle-check">
    The `escalateToHuman` tool pauses AI orchestration and hands off to a human agent. Resume with `resume bot` or `volver al bot`.
  </Card>
  <Card title="RAG knowledge base" icon="database">
    The Docs agent queries the live Botpress documentation sitemap — no static data, always up to date.
  </Card>
  <Card title="needsInput flow" icon="circle-question">
    Specialists signal when they need more context. The orchestrator relays questions to the user without exposing implementation details.
  </Card>
</CardGroup>
